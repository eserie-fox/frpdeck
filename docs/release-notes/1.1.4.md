# frpdeck 1.1.4

`1.1.4` is a patch release focused on client web server control. It adds an explicit `client.web_server.enable` switch so users can keep the default client web server behavior or opt out cleanly without fighting the default `addr` and `port` values.

## Highlights

- Client instances can now disable the rendered frpc `[webServer]` block with `client.web_server.enable: false`.
- Existing client configs keep the previous behavior because `client.web_server.enable` defaults to `true`.
- `reload` now fails fast with a clear message when the client web server is disabled.

## Notable Changes

- Added `client.web_server.enable` to the structured client config model and package defaults.
- Updated client rendering so `[webServer]` is emitted only when `enable`, `addr`, and `port` are all present.
- Kept `enable` as a `frpdeck` configuration switch only; it is not rendered as an FRP TOML field.
- Documented the merge behavior where default `addr` and `port` can remain present while `enable: false` suppresses generated web server config.
- Expanded tests for default config loading, scaffold output, render behavior, and reload failure handling.

## Upgrade Note

No migration is required. Existing client instances continue to render `[webServer]` by default.

To disable the client web server for an instance, add:

```yaml
client:
  web_server:
    enable: false
```

With this setting, `frpdeck render` omits `[webServer]`. The `reload` command will not work for that instance until the client web server is re-enabled, because `frpc reload` depends on the web server control endpoint.
