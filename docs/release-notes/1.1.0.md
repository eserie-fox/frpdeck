# frpdeck 1.1.0

`1.1.0` is a consolidation release. It does not add a large new feature area; instead, it tightens the project's configuration model, logging behavior, scaffold resources, and documentation so the repository is easier to understand, package, and maintain.

## Highlights

- Further converged the project around a single instance-centric configuration model.
- Clarified the difference between FRP's own logging config and `frpdeck`'s logging config.
- Moved more scaffold behavior into package-shipped resources and reduced scattered defaults.
- Tightened config modeling with explicit enum-based log levels and clearer runtime logging setup boundaries.

## Notable Changes

- `frpdeck_logging` remains instance-scoped and separate from FRP `client.log` / `server.log`.
- Scaffold generation now depends on packaged operational defaults, scaffold overrides, and an explicit instance layout resource.
- Runtime logging setup is split into:
  - loading and resolving instance logging config
  - applying an already-resolved logger configuration
- Packaged config resources and project docs were aligned for release and build consistency.

## Upgrade Note

This project is still in a phase where configuration changes are not forward-compatible by default.

If you are carrying forward older instance configs, review:

- top-level `frpdeck_logging`
- current `paths` shape
- scaffold/defaults organization under `config_defaults/`
- the current `docs/configuration.md` reference before reusing existing instances

There is no compatibility layer for pre-`1.1.0` config variants.

## Current Scope

`frpdeck` remains a focused local operations tool for FRP instances:

- instance initialization, validation, rendering, apply/reload/restart/status workflows
- structured proxy management
- append-only audit history and revision snapshots
- local stdio MCP support for instance-bound maintenance

It does not introduce a remote control plane, web UI, or HTTP MCP transport in this release.
